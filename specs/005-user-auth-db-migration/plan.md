# Implementation Plan: User Authentication & Cloud Database with Local Data Migration

**Branch**: `005-user-auth-db-migration` | **Date**: 2026-03-08 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/005-user-auth-db-migration/spec.md`

---

## Summary

Add Supabase Auth (email/password, cookie-based sessions) and a Supabase PostgreSQL cloud database to the SvelteKit personal finance app, replacing the current localStorage-only persistence. Includes an atomic one-time migration flow that transfers existing local wallets, transactions, and custom categories to the authenticated user's cloud account via a PostgreSQL stored procedure. All existing routes become protected; unauthenticated users are redirected to `/login`.

---

## Technical Context

**Language/Version**: TypeScript 5.x / SvelteKit 2.x (existing stack)
**Primary Dependencies**:
- `@supabase/supabase-js` ^2.x — Supabase JS client
- `@supabase/ssr` ^0.x — cookie-based session management for SvelteKit SSR
**Storage**: Supabase PostgreSQL (cloud) — four tables: `wallets`, `transactions`, `categories`, `migration_log`
**Testing**: Vitest (unit), Playwright (E2E) — existing setup
**Target Platform**: Web (mobile-first); deployed on Vercel
**Performance Goals**:
- Login → dashboard: < 3 seconds
- Migration of 1,000 records: < 30 seconds
- Data load on new device after login: < 3 seconds
**Constraints**:
- No `@supabase/auth-helpers-sveltekit` (deprecated)
- `SUPABASE_SERVICE_ROLE_KEY` server-side only, never exposed to browser
- All inserts idempotent (`ON CONFLICT (id) DO NOTHING`)
- Migration is atomic (all-or-nothing via PostgreSQL transaction)
**Scale/Scope**: Personal finance app, single user per account, < 10,000 transactions per user

## Key Technical Decisions

### Auth Integration
- **Package**: `@supabase/ssr` (NOT deprecated `@supabase/auth-helpers-sveltekit`)
- **Session storage**: Cookies (not localStorage) — required for SSR route protection
- **Flow**: PKCE (default in `@supabase/ssr`) — requires `/auth/callback` route
- **JWT validation**: `getUser()` on every request (validates with Supabase servers); `getSession()` used only for client-side data
- **Cookie pattern**: `setAll` (not `set`) — Supabase writes multiple cookies atomically

### Route Protection
- Route group `(app)/` with `+layout.server.ts` guard: throw `redirect(303, '/login?redirectTo=...')` if no `locals.user`
- Route group `(auth)/` with redirect to `/` if user already authenticated
- Auth state sync: `depends('supabase:auth')` in layout server + `invalidate('supabase:auth')` on `onAuthStateChange`

### Database Access
- **Client-side**: `createBrowserClient` in root `+layout.svelte`
- **Server-side**: `createServerClient` fresh per request in `hooks.server.ts`
- **Existing data stores**: Current localStorage stores (`wallets`, `transactions`, `categories`) replaced by Supabase queries after auth is established

### Migration Strategy
- Single PostgreSQL stored procedure `migrate_local_data` called via `supabase.rpc()`
- PostgREST auto-wraps in one transaction → atomic (all or nothing)
- UUID preservation from localStorage → `ON CONFLICT (id) DO NOTHING` for idempotency
- Insert order: categories → wallets → transactions (topological dependency order)
- Migration flag: `raw_app_meta_data.local_migration_completed = true` (service-role-only write, tamper-resistant)
- Detection: client-side in root layout's `$effect` after session confirmed
- Retry: up to 3× with exponential backoff (1s, 3s, 9s) for network errors

### Data Isolation
- RLS enabled on all user tables with `user_id = auth.uid()` policies
- `categories` table: `user_id IS NULL` = global/predefined; user rows have explicit `user_id`
- Soft deletes via `deleted_at TIMESTAMPTZ` — SELECT policies filter `WHERE deleted_at IS NULL`
- CASCADE delete on `auth.users(id)` deletion (GDPR-friendly)
